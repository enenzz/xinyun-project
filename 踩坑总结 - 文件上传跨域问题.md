# 文件上传跨域问题踩坑总结

## 📋 问题背景

在开发图片上传功能时，遇到了一个奇怪的错误：

```
org.springframework.web.servlet.resource.NoResourceFoundException: No static resource 
```

## 🔍 问题现象

### ✅ 正常情况
- 使用 **Apifox** 测试文件上传接口 → **完全正常**
- 后端接口：`POST http://localhost:8080/common/upload`

### ❌ 异常情况
- 使用 **浏览器打开 HTML 文件** 测试上传 → **报错**
- 错误堆栈：
```
org.springframework.web.servlet.resource.NoResourceFoundException: No static resource
    at org.springframework.web.servlet.resource.ResourceHttpRequestHandler.handleRequest(...)
    at org.springframework.web.servlet.mvc.HttpRequestHandlerAdapter.handle(...)
    ...
```

---

## 🎯 问题根源

### 核心原因：跨域请求（CORS）

| 测试方式 | 请求源 | 目标源 | 是否跨域 | 结果 |
|---------|--------|--------|---------|------|
| **Apifox** | - | `http://localhost:8080` | ❌ 否 | ✅ 正常 |
| **浏览器 file://** | `file://` | `http://localhost:8080` | ✅ 是 | ❌ 报错 |

### 详细分析

1. **Apifox 为什么正常？**
   - Apifox 是 API 测试工具，**不受浏览器同源策略限制**
   - 直接发送 HTTP 请求，没有跨域概念

2. **浏览器为什么报错？**
   - HTML 文件使用 `file://` 协议打开
   - 请求 `http://localhost:8080` 属于**跨域请求**
   - 浏览器会先发送 **OPTIONS 预检请求**
   - 后端**没有 CORS 配置**，Spring Boot 将请求当作静态资源处理
   - 找不到对应资源，抛出 `NoResourceFoundException`

3. **为什么会匹配到静态资源处理器？**
   - Spring Boot 的 `ResourceHttpRequestHandler` 会拦截未匹配的请求
   - 跨域请求没有正确的 CORS 响应头
   - Spring Boot 误判为静态资源请求

---

## 💡 解决方案

### 方案一：启用 CORS 配置（推荐）

在 Spring Boot 中添加 CORS 配置：

```java
package com.yunbian.config;

import com.yunbian.interceptor.JwtInterceptor;
import lombok.RequiredArgsConstructor;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.servlet.config.annotation.CorsRegistry;
import org.springframework.web.servlet.config.annotation.InterceptorRegistry;
import org.springframework.web.servlet.config.annotation.WebMvcConfigurer;

@Configuration
@RequiredArgsConstructor
public class WebMvcConfig implements WebMvcConfigurer {

    private final JwtInterceptor jwtInterceptor;

    /**
     * 跨域配置
     */
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/**")
                .allowedOriginPatterns("*")  // 允许所有来源
                .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")  // 允许的请求方法
                .allowedHeaders("*")  // 允许所有请求头
                .allowCredentials(true)  // 允许携带凭证
                .maxAge(3600);  // 预检请求缓存时间
    }

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(jwtInterceptor)
                .addPathPatterns("/**")
                .excludePathPatterns(
                        "/user/login",
                        "/user/register",
                        "/common/**"  // 文件上传接口不需要 JWT 认证
                );
    }
}
```

### 方案二：前端使用 Vite 代理（开发环境）

在 `vite.config.js` 中配置代理：

```javascript
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'

export default defineConfig({
  plugins: [vue()],
  server: {
    proxy: {
      '/api': {
        target: 'http://localhost:8080',
        changeOrigin: true,
        rewrite: (path) => path.replace(/^\/api/, '')
      }
    }
  }
})
```

**优点：**
- 开发环境不跨域
- 不需要后端配置 CORS
- 生产环境切换方便

### 方案三：Nginx 反向代理（生产环境）

```nginx
server {
    listen 80;
    server_name your-domain.com;

    # 前端 Vue 静态文件
    location / {
        root /var/www/html;
        index index.html;
        try_files $uri $uri/ /index.html;
    }

    # 后端 API 反向代理
    location /api/ {
        proxy_pass http://localhost:8080/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        
        # 如果需要，也可以在这里配置 CORS
        # add_header Access-Control-Allow-Origin *;
        # add_header Access-Control-Allow-Methods 'GET, POST, OPTIONS';
    }
}
```

## ✅ 最佳实践建议

### 1. 后端 CORS 配置建议

**即使生产环境用 Nginx 代理，也建议保留 CORS 配置：**

```java
@Override
public void addCorsMappings(CorsRegistry registry) {
    registry.addMapping("/**")
            .allowedOriginPatterns("*")
            .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")
            .allowedHeaders("*")
            .allowCredentials(true)
            .maxAge(3600);
}
```

**理由：**
- ✅ 开发环境需要（Vite、直接测试）
- ✅ 测试工具友好（Apifox、Postman）
- ✅ 生产环境无害（Nginx 代理后不触发）
- ✅ 部署灵活（支持前后端分离部署）

### 2. 前端请求处理

**正确处理后端响应格式：**

```javascript
async function upload() {
    const file = document.getElementById('fileInput').files[0];
    if (!file) {
        alert('请选择图片');
        return;
    }

    const formData = new FormData();
    formData.append('file', file);

    try {
        const response = await fetch('http://localhost:8080/common/upload', {
            method: 'POST',
            body: formData
        });
        
        if (!response.ok) {
            throw new Error(`HTTP error! status: ${response.status}`);
        }
        
        const result = await response.json();
        
        // 后端返回的是 Result 对象：{ code: 200, message: "成功", data: "url" }
        if (result.code === 200 && result.data) {
            const url = result.data;
            document.getElementById('url').innerHTML = '<strong>图片 URL:</strong> ' + url;
            document.getElementById('preview').src = url;
            document.getElementById('result').innerHTML = '上传成功！';
        } else {
            throw new Error(result.message || '上传失败');
        }
    } catch (error) {
        console.error('Upload error:', error);
        document.getElementById('result').innerHTML = '上传失败：' + error.message;
    }
}
```

**关键点：**
- ✅ 检查 `response.ok`
- ✅ 正确解析 `result.data`（不是整个 `result`）
- ✅ 添加错误日志 `console.error`

### 3. 接口路径规范

**避免 URL 末尾斜杠问题：**

```javascript
// ❌ 不推荐（可能被当作目录请求）
fetch('http://localhost:8080/common/upload/')

// ✅ 推荐（明确是接口路径）
fetch('http://localhost:8080/common/upload')
```

## 🚨 常见误区

### 误区 1：Apifox 能通 = 前端也能通

**错误！** Apifox 不受 CORS 限制，浏览器受限制。

### 误区 2：生产环境用 Nginx 就不需要 CORS

**不完全对！** 
- 生产环境确实不需要
- 但开发环境需要
- 保留 CORS 配置对生产环境无害

### 误区 3：前端加 `mode: 'no-cors'` 就能解决

**错误！** `no-cors` 模式会导致：
- 无法读取响应内容
- 无法获取上传后的 URL
- 失去错误处理能力

## 📝 检查清单

在开发文件上传功能时，请检查：

- [ ] 后端已配置 CORS（开发环境需要）
- [ ] 前端正确解析响应数据（`result.data` 不是 `result`）
- [ ] 接口路径没有多余的斜杠
- [ ] 上传接口已排除 JWT 认证（如需要）
- [ ] 前端有完善的错误处理
- [ ] 控制台没有 CORS 相关错误

## 🎓 知识点总结

### 1. 什么是跨域？

**同源策略**：浏览器规定，如果两个页面的**协议、域名、端口**有任何不同，就视为不同源。

**跨域限制**：不同源的页面之间，JavaScript 无法互相访问对方的数据。

### 2. 浏览器如何处理跨域请求？

1. **简单请求**（GET、POST、HEAD）
   - 直接发送请求
   - 检查响应头中的 `Access-Control-Allow-Origin`

2. **预检请求**（POST + 自定义头、PUT、DELETE 等）
   - 先发送 `OPTIONS` 请求
   - 服务器返回允许的 CORS 配置
   - 再发送实际请求

### 3. Spring Boot 静态资源处理

Spring Boot 默认会拦截以下路径作为静态资源：
- `/static/**`
- `/public/**`
- `/resources/**`
- `/webjars/**`

如果请求匹配不到这些路径，又没有被 Controller 处理，就会抛出 `NoResourceFoundException`。

## 💬 总结

这次踩坑的核心教训：

1. **API 工具正常 ≠ 前端正常** - 一定要在浏览器中测试
2. **跨域问题很隐蔽** - 开发环境和生产环境表现不同
3. **CORS 配置要提前** - 不要等报错了再加
4. **错误信息要看全** - `NoResourceFoundException` 背后可能是 CORS 问题

**记住：能用 Apifox 测试通过的接口，在浏览器中不一定能正常工作！**
